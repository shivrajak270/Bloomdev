02 — solutions with annotations
Solution 1 — Sort Map Entries
T: O(n log n)S: O(n)
Pattern: Sort + Slice
Verdict: Simple but suboptimal
Map<Integer,Integer> tmp = new HashMap<>();
// 1. Build frequency map
for(int x : nums)
  tmp.put(x, tmp.getOrDefault(x, 0) + 1);

// 2. Collect entries and sort descending by frequency
List<Map.Entry<Integer,Integer>> list = new ArrayList<>(tmp.entrySet());
list.sort((a, b) -> b.getValue() - a.getValue()); // desc order

// 3. Take first k keys
int[] ans = new int[k];
for(int i = 0; i < k; i++)
  ans[i] = list.get(i).getKey();
return ans;
Solution 2 — Min-Heap of size k
T: O(n log k)S: O(n+k)
Pattern: PriorityQueue / Heap
Verdict: ✔ Classic interview answer
Map<Integer,Integer> h = new HashMap<>();
for(int x : nums) h.put(x, h.getOrDefault(x,0)+1);

// Min-heap: smallest frequency on top
PriorityQueue<Map.Entry<Integer,Integer>> pq =
  new PriorityQueue<>((a,b) -> a.getValue()-b.getValue());

for(Map.Entry<Integer,Integer> e : h.entrySet()){
  pq.add(e);
  if(pq.size() > k) pq.poll(); // evict lowest freq
  // heap never exceeds size k → O(log k) per op
}

int[] ans = new int[k];
int j = 0;
while(!pq.isEmpty()) ans[j++] = pq.poll().getKey();
return ans;
Solution 3 — Bucket Sort (Optimal)
T: O(n)S: O(n)
Pattern: Index = Frequency
Verdict: ✔ Best — impress your interviewer
Map<Integer,Integer> h = new HashMap<>();
for(int x : nums) h.put(x, h.getOrDefault(x,0)+1);

// Bucket index = frequency. Max freq possible = n
List<List<Integer>> buckets = new ArrayList<>();
for(int i = 0; i <= n; i++) buckets.add(new ArrayList<>());

for(Map.Entry<Integer,Integer> e : h.entrySet())
  buckets.get(e.getValue()).add(e.getKey());
  // num with freq 3 → buckets[3]

// Scan right-to-left (highest freq first) until k collected
int[] ans = new int[k]; int j = 0;
for(int i = n; i >= 0 && j < k; i--)
  for(int num : buckets.get(i))
    if(j < k) ans[j++] = num;
return ans;

04 — key concepts

getOrDefault()

Returns the stored value if key exists, otherwise returns the given default. Cleaner than a manual containsKey check: map.getOrDefault(x, 0)+1.

Min-Heap vs Max-Heap

Min-heap keeps the smallest frequency at the top so we can quickly evict it. Using a max-heap would require storing ALL n elements — O(n log n), defeating the purpose.

Bucket Sort Insight

Frequency of any element ∈ [1, n]. So create n+1 buckets indexed by frequency. Elements with the same frequency go in the same bucket. Scan from right (highest freq).

Map.Entry in Java

When you need both key and value together from a HashMap, iterate with map.entrySet(). Each entry exposes .getKey() and .getValue().

Why not int[] as Map key?

Java arrays use reference equality for hashCode(). Two different int[] with same contents will hash differently. Always encode to a String instead.

computeIfAbsent()

Cleaner grouping: map.computeIfAbsent(key, k -> new ArrayList<>()).add(val) — avoids the verbose if(!map.containsKey) pattern entirely.

05 — interview questions & answers
Q1 Why use a min-heap instead of a max-heap for the top-k problem? Easy▼
With a min-heap of size k, the root is always the smallest of the current top-k. When a new element arrives, if its frequency is greater than the root's, we can evict the root and insert the new one — maintaining the invariant in O(log k).

A max-heap would require inserting all n elements first (O(n log n)) and then extracting k times — much worse when k ≪ n.
Q2 Why is bucket sort O(n) here, and not O(n log n)? Easy▼
Bucket sort avoids comparisons. Instead of sorting elements against each other, we use the frequency value as an array index. Since frequencies are bounded integers (1 to n), we can place each element directly into its bucket in O(1). The entire fill + scan is O(n) — no log factor.
Q3 What does getOrDefault() do and why is it preferred? Easy▼
map.getOrDefault(key, defaultValue) returns the mapped value if the key exists, otherwise returns defaultValue without throwing an exception.

It replaces the verbose pattern:
if(map.containsKey(k)) map.put(k, map.get(k)+1); else map.put(k,1);
with a single clean line: map.put(k, map.getOrDefault(k,0)+1);
Q4 Why is bucket size n+1 and not n? Medium▼
The maximum possible frequency of any element is n (when all elements are the same). Bucket indices run from 0 to n — that's n+1 slots. If we only allocated n slots (indices 0 to n-1), accessing buckets.get(n) would throw an IndexOutOfBoundsException.
Q5 What is the space complexity of each solution? Medium▼
  • S1: O(n) — freq map + entry list (both at most n entries)
  • S2: O(n + k) — freq map O(n) + heap O(k)
  • S3: O(n) — freq map O(n) + n+1 buckets (each element appears once)
All are O(n) asymptotically. The output array of size k is typically excluded from space complexity analysis.
Q6 Does HashMap maintain insertion order? What if order matters? Medium▼
No. HashMap makes no guarantees about iteration order. If you need insertion order, use LinkedHashMap. If you need sorted key order, use TreeMap (keys sorted ascending) or TreeMap(Collections.reverseOrder()) for descending.

For this problem, order of the output doesn't matter — the problem only requires the correct k elements, not in any specific order.
Q7 What are the edge cases you must handle? Medium▼
  • k = total unique elements: return all elements — should work naturally
  • nums = [1], k=1: single element, freq=1 → bucket[1] → ans=[1]
  • All elements same: e.g., [3,3,3], k=1 → freq map has one entry → ans=[3]
  • Tie in frequency: problem guarantees the answer is unique, but be aware
  • Negative numbers: handled naturally by HashMap (keys can be any Integer)
Q8 What are common Top-K problem variants using the same heap pattern? Hard▼
The min-heap of size k pattern directly solves:
  • LC 215 — Kth Largest Element in an Array
  • LC 692 — Top K Frequent Words (+ lexicographic tiebreaker)
  • LC 973 — K Closest Points to Origin
  • LC 1337 — K Weakest Rows in a Matrix
  • LC 658 — Find K Closest Elements
  • LC 295 — Find Median from Data Stream (two heaps)
Master heap of size k → solve all of these.
Q9 Top K Frequent Words — how does it differ from this problem? Hard▼
Two differences:
  • Type: keys are Strings, not integers
  • Tiebreaker: if two words have the same frequency, the one that comes first lexicographically ranks higher
Comparator for the min-heap:
(a,b) -> a.getValue().equals(b.getValue()) ? b.getKey().compareTo(a.getKey()) : a.getValue()-b.getValue()

The reversed lexicographic order in the min-heap ensures that when we evict (poll), the lexicographically larger word gets removed first.
Q10 Can you solve this in O(n) without bucket sort? Any other way? Hard▼
Theoretically yes — QuickSelect (partial sort) can find the kth largest in O(n) average time. Steps:
  • Build freq map O(n)
  • Run QuickSelect on the unique keys by their frequency to find the kth pivot O(n) avg
  • Collect all elements ≥ pivot frequency
Worst case is still O(n²) (like QuickSort). Bucket sort is strictly O(n) with no worst case — preferred in interviews. QuickSelect is useful when extra space must be minimized.
