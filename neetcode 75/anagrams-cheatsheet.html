// 02 — solutions with annotations
Solution 1 — Brute Force (Double Loop + Sort)
T: O(n²·k log k) S: O(n)
Pattern: Visited Array
Use when: N is very small
Verdict: Not recommended
// hash[] marks already-grouped strings
for(int i=0; i<n; i++){
  if(hash[i]==1) continue;       // already placed

  List<String> temp = new ArrayList<>();
  char[] first = strs[i].toCharArray();
  Arrays.sort(first);                  // canonical form of strs[i]
  temp.add(strs[i]);
  hash[i] = 1;

  for(int j=i+1; j<n; j++){         // compare with every later str
    if(hash[j]==1) continue;
    char[] second = strs[j].toCharArray();
    Arrays.sort(second);
    if(Arrays.equals(first, second)){  // anagram check
      hash[j] = 1;
      temp.add(strs[j]);
    }
  }
  ans.add(temp);
}
Solution 2 — HashMap with Sorted Key
T: O(n·k log k) S: O(n·k)
Pattern: HashMap Grouping
Use when: General interviews
Verdict: ✔ Recommended default
Map<String, List<String>> map = new HashMap<>();

for(String s : strs){
  char[] ch = s.toCharArray();
  Arrays.sort(ch);                     // "eat","tea","ate" → "aet"
  String key = new String(ch);         // sorted chars = canonical key

  map.computeIfAbsent(key, k ->
        new ArrayList<>())
     .add(s);                          // group under same key
}

return new ArrayList<>(map.values()); // O(1) — just wrap values
Solution 3 — Frequency Count Key (Optimal)
T: O(n·k) S: O(n·k)
Pattern: Frequency Encoding
Use when: k is very large
Verdict: ✔ Optimal (no sort)
// Build unique key: "a2b0c1...z0" from char frequencies
String buildKey(String s){
  int[] freq = new int[26];
  for(char c : s.toCharArray())
    freq[c - 'a']++;            // count each letter

  StringBuilder sb = new StringBuilder();
  char g = 'a';
  for(int cnt : freq){
    sb.append(g).append(cnt); // e.g. "a1e1t1" for "eat"
    g++;
  }
  return sb.toString();        // key is same for all anagrams
}

// Main logic same as S2 — just replace sorted key with freq key
for(String s : strs)
  map.computeIfAbsent(buildKey(s), k -> new ArrayList<>())
     .add(s);
// 03 — key concepts to know

Anagram Definition

Two strings are anagrams if they contain the same characters with the same frequencies. Order doesn't matter. "listen" ↔ "silent".

Canonical Form

A normalized representation shared by all anagrams. Sorting chars gives a canonical form: "eat", "tea", "ate" → "aet".

HashMap Grouping

Use a Map<String, List<String>> where key = canonical form. All anagrams land in the same bucket. O(1) average insert & lookup.

Why Freq Key is Faster

Sorting is O(k log k). Counting frequencies is O(k). For long strings (large k), the frequency key approach wins. Both have same space complexity.

computeIfAbsent

Clean Java idiom: map.computeIfAbsent(key, k → new ArrayList<>()).add(val). Avoids manual containsKey check.

Edge Cases

Single string input, all strings identical, empty strings "" (group together), strings with length 1, mixed-case (problem usually guarantees lowercase).

// 04 — interview questions & answers
Q1 What is the time complexity of Solution 2 and why? Easy ▼
O(n · k log k) where n = number of strings, k = max string length.
  • For each of the n strings, we sort its characters: O(k log k).
  • HashMap insert/lookup is O(k) due to string hashing (key length is k).
  • Total: O(n·k log k). Space is O(n·k) for storing all strings in the map.
Q2 Why is Solution 3 better than Solution 2 asymptotically? Medium ▼
S3 avoids sorting by building a frequency-based key in O(k) instead of O(k log k). For very long strings, this matters significantly.

Both use the same HashMap approach — the only difference is how the key is built. S3's total complexity is O(n·k), which is the theoretical optimum (you must read every character at least once).
Q3 What makes a good grouping key? Could you use int[26] directly as a HashMap key? Medium ▼
No, you cannot use int[] directly as a HashMap key in Java. Arrays use reference equality (==) for hashCode() and equals(), not content equality. Two different int[] arrays with identical contents will hash differently.

Instead, encode the frequency array into a String (as in S3) or use Arrays.toString(freq). Strings in Java have proper content-based hashing.
Q4 Why is Solution 1 not suitable for production/interviews at scale? Easy ▼
S1 uses a nested loop (O(n²)) and sorts inside the inner loop, giving O(n²·k log k). For n = 10⁴ strings, this is ~10⁸ operations — likely TLE on LeetCode and any production system with large datasets.

The fix is to use a HashMap (S2/S3), reducing the outer iteration to a single pass O(n).
Q5 How would you handle unicode/uppercase characters? Medium ▼
  • Uppercase: Normalize with s.toLowerCase() before processing if case-insensitive grouping is needed.
  • Unicode: The int[26] approach only works for lowercase a–z. For unicode, use a HashMap<Character, Integer> for frequency counting, then encode it into a sorted string of (char, count) pairs.
  • The sorting approach (S2) naturally handles any character set since Arrays.sort works on any comparable chars.
Q6 Can you solve this without extra space (in-place)? Hard ▼
Not truly in-place for the output — we must return grouped lists, which requires O(n·k) output space regardless. However, we can minimize auxiliary space:
  • S1 uses O(n) auxiliary space (hash[] array), but still O(n·k) total output.
  • S2/S3 use O(n·k) for the HashMap (keys + values).
  • If output space is excluded, S1 is "most space-efficient" in auxiliary terms, but at the cost of O(n²) time.
In an interview, clarify whether "space" means total or auxiliary before optimizing.
Q7 What data structure variation would you use if input strings keep streaming in? Hard ▼
For a streaming/online variant, maintain a persistent HashMap<String, List<String>> where key = canonical form (sorted or frequency-encoded).

Each new string is processed in O(k log k) or O(k) and inserted into the map. Querying for a group is O(k) for key generation + O(1) lookup. This is effectively what S2/S3 do but can be applied incrementally. You could expose an addString(s) and getGroups() API.
Q8 How would you write a unit test for groupAnagrams? Easy ▼
Key test cases to cover:
  • ["eat","tea","tan","ate","nat","bat"] → 3 groups
  • [""] → one group with empty string
  • ["a"] → one group with "a"
  • All identical strings → one group
  • No anagrams → each string in its own group
  • Large input (n = 10⁴) for performance
Since output order is non-deterministic, sort each inner list and the outer list before asserting equality.
