🧠 Q1. How does your algorithm work? Explain the approach. MUST ▼

For every element compute complement = target - nums[i]. Check if complement is already in the HashMap. If yes → return both indices. If no → store current number and its index, continue.

public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> map = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (map.containsKey(complement)) {
            return new int[]{ map.get(complement), i };
        }
        map.put(nums[i], i);
    }
    throw new IllegalArgumentException("No solution");
}
💡 Say this out loud: "I use a HashMap so I don't need two loops. For each number I check if its complement already exists. One pass is enough." — Say the logic FIRST, then write code.

💾 Q3. What is the space complexity? MUST ▼

HashMap stores up to n elements → O(n) space.

Brute force needs no extra memory → O(1) space, but O(n²) time.

This is a space-time tradeoff — we spend memory to save time.

🔑 Q4. Why check containsKey BEFORE put? Trace nums=[3,3], target=6. MUST ▼

If you put first, the second 3 overwrites the first one's index. Then containsKey returns the same index twice — which is wrong.

✅ CORRECT — check first, then put:
i=0 → diff=3, map empty  → map.put(3, 0)    map={3:0}
i=1 → diff=3, found at 0 → return {0, 1}   ✅

❌ WRONG — put first, then check:
i=0 → map.put(3, 0)                          map={3:0}
i=1 → map.put(3, 1)  ← overwrites index 0!  map={3:1}
     → containsKey(3) → returns {1, 1}  ❌ same index!
💡 Draw this trace on paper or whiteboard when they ask. Visualising = looks very sharp.
↔️ Q5. What if the array is sorted? (Two Sum II) MUST ▼

Use Two Pointers. Start from both ends, move inward based on the sum.

int left = 0, right = nums.length - 1;
while (left < right) {
    int sum = nums[left] + nums[right];
    if      (sum == target) return new int[]{ left, right }; // found
    else if (sum < target)  left++;   // need bigger
    else                    right--;  // need smaller
}

Time: O(n)   Space: O(1) — no HashMap needed at all.

💡 Always ask first: "Is the array sorted?" before jumping to HashMap. It changes the best approach completely.

⚙️ Q8. How does HashMap work internally? GOOD TO KNOW ▼

🗂 Step 1 — What is a Bucket?

Think of a HashMap like a big shelf with 16 compartments (called buckets). When you store something, Java figures out which compartment to put it in. Default shelf size = 16 buckets.

// Imagine this shelf:
Bucket[0]  → empty
Bucket[1]  → empty
Bucket[2]  → "apple" → 5
Bucket[3]  → empty
Bucket[4]  → "mango" → 3
...
Bucket[15] → empty

🔢 Step 2 — hashCode() decides the bucket

When you call map.put("apple", 5), Java runs "apple".hashCode() which gives a big number. Then it does bigNumber % 16 to get a bucket index between 0–15.

// Simplified flow of map.put("apple", 5):
1. hash  = "apple".hashCode()   // e.g. 93029210
2. index = hash % 16           // e.g. 2  ← bucket number
3. Bucket[2] = "apple" → 5    // stored here

// map.get("apple") does the SAME thing:
1. hash  = "apple".hashCode()   // same hash = same bucket
2. index = hash % 16           // = 2
3. Go to Bucket[2], return 5  // O(1) ✅

That's why get/put is O(1) — no matter how many items you have, Java directly jumps to the right bucket instead of searching through everything.

💥 Step 3 — What is a Collision?

Sometimes two different keys land in the same bucket. This is called a collision. For example, "apple" and "grape" might both hash to Bucket[2].

// Both "apple" and "grape" land in Bucket[2]:
Bucket[2] → "apple"=5  →  "grape"=9  →  null
           ↑ linked list — each node points to the next

// map.get("grape") now has to:
1. Go to Bucket[2]
2. Check first node: is it "grape"? No, it's "apple". Skip.
3. Check next node: is it "grape"? Yes! Return 9.  // O(n) worst case

If 1000 keys all crash into the same bucket, you'd have to scan all 1000 nodes = O(n). Very slow.

🚀 Step 4 — Java 8 Fix: Linked List → Red-Black Tree

Java 8 solved this. When a single bucket's linked list grows beyond 8 nodes, Java automatically converts it to a Red-Black Tree (a self-balancing tree). Searching a tree is O(log n) instead of O(n).

// Java 7 (old) — all collisions use Linked List:
Bucket[2] → node1 → node2 → node3 → ... → node1000
           worst case lookup = O(n)  ❌ slow

// Java 8 (new) — long chains become Red-Black Tree:
Bucket[2] → [ Red-Black Tree with 1000 nodes ]
           worst case lookup = O(log n)  ✅ fast

// Trigger: chain length > 8  →  convert to tree
// Reverse:  chain length < 6  →  convert back to list

You don't need to do anything special — Java does this automatically inside HashMap.

📈 Step 5 — What is Resize (Rehashing)?

HashMap doesn't stay 16 buckets forever. When it's 75% full (12 out of 16 slots used), it doubles to 32 buckets, then recalculates every key's bucket. This is called rehashing.

// Load Factor = 0.75 (default)
// Capacity starts at 16

16 buckets × 0.75 = 12  ← resize triggers when 12 entries added
→ new capacity = 32 buckets
→ all keys rehashed into new positions

32 × 0.75 = 24  ← next resize at 24 entries
→ new capacity = 64 buckets  ... and so on

Resize is expensive (O(n)) but rare. Overall HashMap stays O(1) average.

OperationAverage CaseWorst Case (Java 7)Worst Case (Java 8+)

resizeO(n) — but happens rarely, amortized O(1)
💡 The one sentence that impresses interviewers: "Java 8 upgraded HashMap to use a Red-Black Tree instead of a Linked List when a bucket has more than 8 collisions — making worst-case lookup O(log n) instead of O(n)." — Say this confidently and they will remember you.
🚫 Q9. Can you solve it without extra space? GOOD TO KNOW ▼

Yes — two options, no HashMap needed:

  • Brute force: two nested loops → O(n²) time, O(1) space
  • Sort + Two Pointers: O(n log n) time, O(1) space — but original indices are lost after sorting
💡 If they say "no extra memory" → say "Sort + Two Pointers" immediately, and add "but this loses the original indices — is that okay?" Shows you understand the full tradeoff.
📦 Q10. What if there are multiple pairs? Return all of them. GOOD TO KNOW ▼

Don't return early. Collect all results into a List. Sort + Two Pointers handles duplicate-skipping cleanly:

public List<int[]> allPairs(int[] nums, int target) {
    Arrays.sort(nums);
    List<int[]> res = new ArrayList<>();
    int l = 0, r = nums.length - 1;
    while (l < r) {
        int sum = nums[l] + nums[r];
        if (sum == target) {
            res.add(new int[]{ nums[l], nums[r] });
            while (l < r && nums[l] == nums[l+1]) l++; // skip duplicates
            while (l < r && nums[r] == nums[r-1]) r--;
            l++; r--;
        } else if (sum < target) l++;
        else r--;
    }
    return res;
}
💡 Bonus: This is the exact same pattern used in 3Sum. Learn this once, apply it everywhere.
