Q 01

Why is String immutable in Java?

VERY HIGH PROB
  • Security 🔐 — Strings hold passwords, DB URLs, file paths, class names. If mutable, a method with a reference could silently modify critical data like jdbc:mysql://localhost mid-execution.
  • String Pool Memory Optimization — JVM reuses one pool object across many references. This only works safely because nobody can mutate it — a change would corrupt all other references pointing to the same pool entry.
  • Thread Safety — Free of Cost — Multiple threads can read the same String simultaneously with zero locks. No mutation = no race condition. Zero synchronization overhead.
  • Stable HashCode for HashMap / HashSet — String caches its hash after first computation. If content could change, the cached hash would go stale → broken map lookups, lost keys. Immutability guarantees hash stability forever.
⚙
How Java enforces it: The internal char[] value field is private final. No setter exists. Every method like toUpperCase(), concat() returns a brand-new String — the original is never touched.
💬
Perfect interview answer: "String is immutable in Java to ensure security, thread safety, memory optimization through the string pool, and stable hashcodes for collections like HashMap."

Q 04

What is the String Constant Pool?

MEMORY
  • A deduplication cache inside the JVM Heap that stores one copy of each unique String literal. Multiple references share the same object safely — because it cannot be mutated.
  • How it works: JVM sees "java" → checks pool. Found → return existing ref. Not found → create entry, return ref. Called string interning.
  • Memory saving: In large apps, the same strings appear millions of times. The pool stores exactly one copy per unique value.
PoolDemo.java
String a = "java";  // adds "java" to pool @ 0x100
String b = "java";  // finds "java" in pool → returns 0x100
a == b   → true   // same reference, zero extra memory
📦
Java 7+ change: Pool moved from PermGen (fixed size, caused OutOfMemoryError) to the main Heap (garbage-collected). This fixed a notorious production crash in older apps.

Q 06

What is intern()?

API
  • intern() manually adds a String to the pool (if absent) and returns the canonical pool reference. After this, == comparison works safely.
  • Real-world use: Deduplicating millions of repeated strings (country codes, status values) from DB to reduce heap pressure.
InternDemo.java
String s1 = new String("hello");  // heap @ 0x200
String s2 = s1.intern();          // returns pool ref @ 0x100
String s3 = "hello";             // pool ref @ 0x100

s1 == s2   → false   // heap vs pool
s2 == s3   → true    // both pool @ 0x100 ✓
Q 07

Why is StringBuilder faster than String +=?

PERFORMANCE
❌ BAD — O(n²)
String s = "";
for(int i=0; i<1000; i++){
    s = s + i;  // new object each time
}
// ~1000 String objects
// O(n²) total time
✅ GOOD — O(n)
StringBuilder sb = new StringBuilder();
for(int i=0; i<1000; i++){
    sb.append(i); // same buffer
}
// 1 object, O(n) time ✓
  • String: Each + allocates new char array, copies old content, appends. O(n) per op → O(n²) in loop.
  • StringBuilder: Internal char[] doubles when full (like ArrayList). Each append() is O(1) amortized. No copy, no new object.

Q 09

Why is String used as a HashMap key instead of StringBuilder?

IMPORTANT
  • HashMap uses hashCode to find the bucket for a key. If the key changes after insertion, its hash changes → bucket mismatch → lookup fails, value is "lost".
  • String is immutable → hashCode is computed once and cached. It never changes. Map lookups always work correctly.
  • StringBuilder is mutable → you can call .append() after putting it in the map → its content and hash change → the map can no longer find that entry.
Why StringBuilder breaks maps
HashMap<StringBuilder,Integer> map = new HashMap<>();
StringBuilder key = new StringBuilder("apple");
map.put(key, 10);

key.append("fruit");       // key is now "applefruit"
map.get(key);              → null // hashCode changed! bucket mismatch
💬
Perfect answer: "String is immutable so its hashCode never changes. HashMap relies on consistent hashCodes to locate keys. Using a mutable key like StringBuilder could change hashCode after insertion, making the entry unretrievable."

Q 12

Why not use StringBuilder everywhere?

IMPORTANT
  • Not thread-safe — Two threads appending to the same StringBuilder simultaneously can corrupt its internal buffer silently. Use StringBuffer in multi-threaded scenarios.
  • Cannot be used as HashMap key — Mutable hashCode would break the map (see Q9).
  • Not stored in String Pool — No memory deduplication benefit. Every new StringBuilder() is a fresh heap object.
  • Verbose for simple fixed values — String country = "India" is simpler and safer than a StringBuilder when the value never changes.

ADV 01

How is String hashCode calculated internally?

ADVANCED
  • Java uses the polynomial rolling hash formula: hash = s[0]×31^(n-1) + s[1]×31^(n-2) + ... + s[n-1]
  • Simplified: for each character, hash = 31 * hash + char
  • Result is cached in the hash field after first call → subsequent hashCode() calls are O(1), not O(n).
HashCode formula
// Simplified internal implementation:
int hash = 0;
for (char c : value) {
    hash = 31 * hash + c;
}
// Result is cached → hashCode() is O(1) on subsequent calls
🔢
Why 31? It's a prime number, which minimizes hash collisions. Also, 31 * x can be computed as (x << 5) - x — a bit shift optimization the JVM can exploit for speed.
ADV 02

What does Arrays.sort() use internally?

ADVANCED
Data TypeAlgorithmWorst CaseWhy
Primitive arrays (int[], char[])Dual-Pivot QuickSortO(n²) worstIn-place, fast in practice
Object arrays (String[], Integer[])TimSortO(n log n)Stable, great on real data
  • Dual-Pivot QuickSort — Uses 2 pivots instead of 1, reducing comparisons. Very fast on random data. Not stable (equal elements may reorder).
  • TimSort — Hybrid of Merge Sort + Insertion Sort. Detects already-sorted runs and merges them. Stable — preserves order of equal elements. Python also uses TimSort.
🤔
Interview question: "Why not QuickSort for objects?" → Because QuickSort is unstable. For objects with equal fields (e.g., two Students with same marks), QuickSort may arbitrarily swap them. TimSort is stable, preserving original order.
ADV 03

Unicode length trap — emoji.length()

TRAP
UnicodeTrap.java
String s = "😊";
s.length()           → 2   // NOT 1 !
s.codePointCount(0, s.length())  → 1   // actual character count
  • Java uses UTF-16 encoding internally. Most characters fit in one 16-bit char.
  • Emoji and some rare Unicode characters (called supplementary characters) require 2 chars (a surrogate pair). So length() returns 2 for a single emoji.
  • Use codePointCount(0, s.length()) to get the real number of visible characters.
ADV 04

substring() memory leak (pre-Java 7)

ADVANCED
  • Before Java 7u6: substring() shared the original char[] array. A tiny substring of a 100MB string would keep the entire 100MB array alive in memory — a classic memory leak.
  • After Java 7u6 (current behavior): substring() copies only the relevant characters into a new char[]. The original can be garbage collected. Problem solved.
📚
Why it matters now: Interviewers sometimes ask this to check if you know Java version history. If on an old JVM, prefer new String(original.substring(...)) to force a copy and release the original.

T-03

Compile-time constant vs runtime concatenation

TRAP
ConcatTrap.java
String a = "hello";
String b = "hel" + "lo";    // compile-time constant → pool ✓
String c = "hel";
String d = c + "lo";      // runtime concat → new heap object

a == b    → true    // both compile-time → same pool ref
a == d    → false   // d built at runtime → heap object
⚙
Why b is pool: The compiler evaluates "hel" + "lo" at compile time (constant folding) and replaces it with "hello". But when a variable is involved in +, it's a runtime operation — a new heap object is created.

String.valueOf(null) vs null.toString()

TRAP
NullString.java
String.valueOf(null)   → "null"    // static method, handles null safely
(String) null          → null     // valid cast, still null reference
null.toString()        → NPE!     // can't call instance method on null
"abc".equals(null)    → false    // safe, returns false
null.equals("abc")    → NPE!     // calling on null reference
✅
Safe pattern: Always put the known non-null string on the left: "expected".equals(variable) instead of variable.equals("expected"). This way, even if variable is null, no NPE occurs.

P 05

Longest Substring Without Repeating Characters

SLIDING WINDOW
LongestSubstring.java — O(n)
public int lengthOfLongestSubstring(String s) {
    Set<Character> set = new HashSet<>();
    int left = 0, max = 0;

    for(int right=0; right<s.length(); right++){
        while(set.contains(s.charAt(right))){
            set.remove(s.charAt(left++));  // shrink window
        }
        set.add(s.charAt(right));
        max = Math.max(max, right - left + 1);
    }
    return max;
}
🪟
Sliding Window pattern: Two pointers (left, right). Expand right. When duplicate found, shrink from left until duplicate removed. Track max window size throughout.
P 04

First Non-Repeating Character

FREQ ARRAY
FirstUnique.java — O(n)
public int firstUniqChar(String s) {
    int[] freq = new int[26];

    for(char c : s.toCharArray())
        freq[c - 'a']++;

    for(int i=0; i<s.length(); i++)
        if(freq[s.charAt(i) - 'a'] == 1)
            return i;

    return -1;
}
